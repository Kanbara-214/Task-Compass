# 技術概要

Task Compass の技術構成、設計方針、テスト方針をまとめます。

## 技術構成

- バックエンド: Java 21, Spring Boot
- 認証・認可: Spring Security
- テンプレート: Thymeleaf
- DB アクセス: MyBatis
- データベース: H2, PostgreSQL
- バリデーション: Jakarta Bean Validation
- テスト: JUnit 5, Spring Boot Test, MockMvc
- CI: GitHub Actions

## 技術選定理由

Java と Spring Boot は、Java Gold の学習内容を実際の Web アプリケーション開発に接続しやすく、求人でも使用例が多いため採用しました。

Spring Security は、ログイン機能だけでなく、認証済みユーザーごとのアクセス制御を実装するために使用しています。ポートフォリオとして、単に画面を作るだけでなく、他ユーザーのデータにアクセスできないことまで示したかったためです。

Thymeleaf は、バックエンド中心の構成で画面まで一通り作れるため採用しました。React などの SPA 構成も候補でしたが、今回は Java / Spring の理解を優先し、サーバーサイドレンダリングに寄せました。

MyBatis は、SQL を明示的に書きながら DB アクセスを理解するために選びました。JPA も候補でしたが、学習段階では発行される SQL が見えやすい構成の方が、データ取得条件や ownerId による制御を説明しやすいと判断しました。

H2 はローカル起動の手軽さを優先して採用し、PostgreSQL は本番に近い DB を使える余地を残すために対応しました。

## AI 生成から見直した点

初期実装は AI 生成をベースにしていますが、そのまま完成扱いにはせず、以下の点を見直しました。

- 認証・認可まわりの挙動を確認し、未認証アクセスや他ユーザーのタスクアクセスをテストで検証
- `TaskNotFoundException` を導入し、存在しないタスクや権限のないタスクを 404 として扱うように修正
- form、query、model などのパッケージを整理し、責務が分かる構成に変更
- README と docs を分け、設計意図や現在の課題を説明できる形に整理

特に、他ユーザーのタスクを閲覧・更新・削除できないことは、ポートフォリオでも重要な観点だと考え、テストを追加しました。

## なぜこの設計にしたか

このアプリでは、タスクが必ずログインユーザーに紐づくため、データ取得時に `taskId` だけでなく `ownerId` を条件に含める設計にしました。

また、優先度スコアの計算は画面や Controller に置かず、`PriorityScoringService` に分離しています。これにより、スコア計算の変更やテストをしやすくしています。

画面表示では、DB エンティティをそのままテンプレートへ渡すのではなく、`TaskView` や `DashboardView` を使っています。表示用の文字列、優先度情報、集計結果を Service 側で整理することで、テンプレートが複雑になりすぎないようにしました。

## 悩んだ点

優先度スコアをどの程度まで複雑にするかは悩みました。

最初からカテゴリ別の重みづけやユーザーごとの設定を入れることも考えましたが、まずは「なぜ上位に表示されているか」を説明できるシンプルなルールに寄せました。

また、一覧画面の絞り込み・並び替えを SQL 側で行うか、Java 側で行うかも悩みました。現状は実装の見通しを優先して Java 側で処理していますが、データ件数が増える前提では DB 側に寄せるべきだと考えています。

## 捨てた選択肢

SPA 構成は採用しませんでした。フロントエンドを React などで分離すると見栄えや操作性の幅は広がりますが、今回は Java / Spring / DB / 認証 / テストを一通り示すことを優先しました。

JPA も採用しませんでした。エンティティ中心で実装できる利点はありますが、今回は SQL と ownerId 条件を明示し、データ分離の意図をコード上で追いやすくするため MyBatis を選びました。

最初から高度なレコメンド機能にすることも避けました。機械学習や履歴ベースの最適化ではなく、入力値から説明可能なルールで優先度を出す方が、このアプリの規模では妥当だと判断しました。

## アプリケーション構成

Spring Boot を使ったサーバーサイドレンダリング構成です。

Controller でリクエストを受け取り、Service で業務ロジックを扱い、Mapper 経由で DB にアクセスします。画面表示は Thymeleaf テンプレートで行います。

## パッケージ構成

- `config`: Spring Security、共通 ViewModel、デモデータ投入
- `controller`: 画面リクエストの受付
- `entity`: DB テーブルに対応するモデル
- `exception`: アプリケーション固有の例外
- `form`: 入力フォームとバリデーション
- `mapper`: MyBatis による DB アクセス
- `model`: 画面表示用モデル
- `query`: 一覧検索条件
- `security`: Spring Security 用の認証ユーザーモデル
- `service`: 業務ロジック

## 認証・認可

Spring Security を使って、ログイン済みユーザーのみがタスク画面へアクセスできるようにしています。

未認証ユーザーがタスク一覧、作成、詳細、編集画面にアクセスした場合はログイン画面へリダイレクトします。

## ユーザー単位のデータ分離

タスクは `owner_id` でユーザーに紐づけています。

取得・更新・削除では `taskId` だけでなく `ownerId` も条件に含め、URL を直接変更しても他ユーザーのタスクを閲覧・編集・削除できないようにしています。

## Controller / Service / Mapper の責務

Controller はリクエストパラメータの受け取り、認証ユーザーの取得、View への値渡しを担当します。

Service はタスク作成、更新、削除、ダッシュボード集計、優先度計算との連携など、アプリケーションの中心となる処理を担当します。

Mapper は SQL を使った DB アクセスを担当します。SQL を明示的に書くことで、どの条件でデータを取得・更新しているかを確認しやすくしています。

## 画面表示用モデル

DB エンティティをそのまま画面に渡すのではなく、`TaskView`、`DashboardView`、`PriorityInsight` などの画面表示用モデルを用意しています。

これにより、テンプレート側で複雑な計算を行わず、表示に必要な値を Service 側で整理してから渡す構成にしています。

## バリデーション

タスク作成・編集では、タイトル、締切、重要度、緊急度、予想作業時間、カテゴリなどに入力制約を設けています。

不正な入力がある場合は保存せず、入力フォームに戻してエラーを表示します。

## 異常系ハンドリング

存在しないタスクや、ログインユーザーが所有していないタスクにアクセスした場合は `TaskNotFoundException` を投げ、404 として扱います。

権限がないタスクに対しても「存在しないタスク」と同じ応答にすることで、他ユーザーのタスク ID の存在を推測しにくくしています。

## データベース設計

主なテーブルは `app_users` と `tasks` です。

`app_users` ではユーザー情報を管理し、メールアドレスには一意制約を設けています。

`tasks` ではタスク情報を管理し、`owner_id` でユーザーに紐づけています。重要度、緊急度、予想作業時間には DB 側でも CHECK 制約を設けています。

検索・並び替えで使う `owner_id, updated_at` と `owner_id, due_date` にはインデックスを作成しています。

## テスト方針

Controller では MockMvc を使い、画面遷移、認証制御、他ユーザーのタスクに対するアクセス制御、バリデーションエラー時の保存抑止を確認しています。

Service では、優先度スコアの代表的なケースをテストしています。

特に、他ユーザーのタスクを閲覧・更新・削除できないこと、存在しないタスクに 404 を返すことを重点的に確認しています。

## CI

GitHub Actions で `push` と `pull_request` 時に `./mvnw test` を実行します。

Java 21 のセットアップと Maven キャッシュを利用し、テストが通る状態をリモートでも確認できるようにしています。

## 現在の課題と今後の改善

- 一覧の絞り込み・並び替えを Java 側で行っており、データ件数が増えた場合に効率が落ちる
- 優先度スコアの重みづけが固定値で、ユーザーごとの調整はできない
- ページネーションがなく、タスク数が増えたときの表示に課題がある
- 現時点では公開デプロイ環境がない

現在の構成は、小規模な個人利用を想定したシンプルな設計です。タスク数が増えた場合は、一覧検索条件を Mapper に渡し、DB 側で絞り込み・並び替えを行う必要があります。

また、優先度スコアは固定ルールのため、ユーザーごとの価値観や作業傾向までは反映できません。今後は、スコアの重みづけ設定、カテゴリごとの優先度補正、締切なしタスクや繰り返しタスクへの対応を検討したいです。

運用面では、ページネーションの追加とデプロイ環境の用意が次の課題です。
